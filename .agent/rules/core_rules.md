# WHMCS Core (vnx-whmcs-v3) Development Rules

Rules for developing the WHMCS core source code, global hooks, and main templates.

## 1. Security First
- Every PHP file MUST start with the WHMCS access check:
  ```php
  if (!defined("WHMCS")) {
      die("This file cannot be accessed directly");
  }
  ```
- Use WHMCS `Capsule` for all database operations. Never use raw `mysql_` or `mysqli_` functions.
- Always validate and sanitize user input from `$_POST`, `$_GET`, and `$_REQUEST`.

## 2. Core Development (Global)
- **Global Hooks**: Place global hooks in `includes/hooks/`. Use descriptive filenames (e.g., `vietnix_custom_behavior.php`).
- **Cron Jobs**: Place custom background tasks in `crons/`. Follow the existing pattern of including `init.php` and checking for CLI environment if necessary.
- **Root Files**: Avoid adding new files to the root directory unless absolutely necessary. Prefix custom root files with `vnx` (e.g., `vnxajax_custom.php`).
- **Includes**: Custom global logic that doesn't belong to a module should be placed in `includes/` or a dedicated subdirectory within `includes/`.
- **Formatting**: Maintain consistency with the existing codebase (typically PSR-12 or similar PSR standards).

## 3. UI/Template Quality Control
- **Smarty Syntax**: Ensure all `{` and `}` tags are balanced. Use `{literal}` tags for inline JavaScript or CSS that contains braces to avoid Smarty parsing errors.
- **Tailwind CSS & Webpack Mix**: In `vietnix_v3` and `orderforms`, ensure `yarn dev` runs without errors. This validates that Tailwind and Webpack Mix are correctly configured.
- **Responsive Design**: All templates MUST be responsive. Use Tailwind's utility classes (`sm:`, `md:`, `lg:`) for layout adjustments.
- **Path Consistency**: Use WHMCS template variables like `{$WEB_ROOT}`, `{$template}`, and `{$assetPath}` for linking assets to ensure they work in all environments.
- **Template Debugging**: If a UI error occurs, check the WHMCS Admin -> System Logs -> Utilities -> Logs -> Smarty Error Log (if enabled).

## 4. Asset Building & Tooling
- Use **Yarn** exclusively for managing JavaScript/CSS dependencies and build scripts. Do not use npm.
- Use `yarn dev` to build assets for development.
- Do not commit `node_modules/` or built assets that are generated by the build process.
