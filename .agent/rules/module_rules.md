# WHMCS Module Development Rules (Vietnix Standards)

Rules for developing WHMCS Server and Addon modules.

## 1. Security First
- Every PHP file MUST start with the WHMCS access check:
  ```php
  if (!defined("WHMCS")) {
      die("This file cannot be accessed directly");
  }
  ```
- Use WHMCS `Capsule` for all database operations. Never use raw `mysql_` or `mysqli_` functions.
- Always validate and sanitize user input from `$_POST`, `$_GET`, and `$_REQUEST`.

## 2. Directory Structure & Architecture

### Server Modules
- Path: `modules/servers/[ModuleName]/`
- Main file: `[ModuleName].php`
- Use `composer.json` for PSR-4 autoloading (Namespace: `VMWS\`).
- Logic should be encapsulated in classes within the `class/` or `lib/` directory.
- Templates should be in `templates/`.

### Addon Modules
- Path: `modules/addons/[ModuleName]/`
- Main file: `[ModuleName].php`
- Follow MVC-like pattern:
  - `Model/`: Database interactions.
  - `Api/`: External API calls.
  - `Lib/`: Core business logic and Dispatchers.
  - `templates/`: Smarty `.tpl` files.
- Use `composer.json` for PSR-4 autoloading (Namespace: `VMW\`).

## 3. Coding Standards
- Use CamelCase for Classes and camelCase for methods/variables.
- WHMCS Hook functions must follow the naming convention: `[ModuleName]_[HookName]`.
- Module core functions (e.g., `_ConfigOptions`, `_MetaData`, `_CreateAccount`) must be prefixed with `[ModuleName]_`.
- Use `logModuleCall()` for debugging API interactions with external services.
- Use `logActivity()` for logging important system events.

## 4. Dependencies
- Always include `require_once __DIR__ . '/vendor/autoload.php';` in the main module file.
- Do not commit the `vendor/` directory; provide instructions for `composer install`.

## 5. UI/UX (Vietnix Premium)
- Use Smarty templates for all output.
- Pass variables to templates using an associative array.
- Organize assets (JS, CSS, Images) in `templates/public/[js|css|img]`.

## 6. Testing Standards
- Use **PHPUnit** for all unit and functional tests.
- Place tests in the `tests/` directory of the module.
- Mock all WHMCS global functions and classes (e.g., `Capsule`, `logModuleCall`) using the provided `WhmcsMock` helper or standard PHPUnit mocks.
- Every major business logic method in `Lib/` or `class/` SHOULD have a corresponding test case.
- Test filenames MUST end with `Test.php` (e.g., `TemplateTest.php`).
- Run tests regularly using `vendor/bin/phpunit`.

## 7. Asset Building & Tooling
- Use **Yarn** exclusively for managing JavaScript/CSS dependencies and build scripts. Do not use npm.
- Use `yarn dev` to build assets for development (e.g., compiling Sass, bundling JS with Webpack/Vite).
- Ensure `package.json` includes at least the `dev` script.
- Do not commit `node_modules/` or built assets that are generated by the build process.
